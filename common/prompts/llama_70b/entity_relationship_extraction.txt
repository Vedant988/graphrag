# Knowledge Graph Extraction Instructions

You are extracting a high-recall, retrieval-oriented knowledge graph from the input text.
Your job is to preserve factual distinctions, recover explicit relationships, and retain enough semantic payload that downstream GraphRAG can answer questions with minimal raw text.

## 1. Core Objective
- Prioritize fidelity over simplicity.
- Do not merge distinct entities just to make the graph look cleaner.
- Preserve specific actors, roles, events, requests, kinship ties, causal links, and narrative actions when they are stated or clearly implied by the passage.
- Prefer a partial but faithful graph over an over-simplified graph.

## 2. Entity Extraction Rules
- Extract all relationally important entities, even if they appear only once.
- Use human-readable string IDs only. Never use integers as entity IDs.
- Keep entity IDs as close as possible to the text while resolving ambiguity when needed.
- When the same surface form refers to different things, create separate contextualized IDs.
  Example: `Trunk (Mahabharata Epic)` and `Trunk (Yudhishthira Metaphor)`.
- Do not merge entities unless the passage clearly indicates they are the same referent.
- Resolve pronouns and aliases to the most specific unambiguous referent when possible.

## 3. Entity Typing Rules
- Use the most specific stable node type supported by the text.
- Do not collapse all humans into `person` if a more meaningful type is available.
- Prefer types such as:
  `Person`, `Warrior`, `Sage`, `Deity`, `RoyalFigure`, `Group`, `Place`, `Event`, `TextWork`, `Artifact`, `Lineage`, `Concept`.
- If none fit well, create a concise domain-relevant type rather than using an overly generic fallback.

## 4. Node Definition Rules
- Every extracted node should have a useful `definition`.
- The definition should briefly describe who or what the entity is in the context of the input passage.
- Keep definitions factual and grounded in the passage.
- Do not create extra nodes for every descriptive adjective, metaphor, or abstract phrase.
- Only promote a concept to its own node if it is independently referential, participates in relationships, or is important for retrieval.

## 5. Relationship Extraction Rules
- Extract relationships exhaustively.
- If a sentence or list expresses multiple distinct facts, create one relationship per fact.
- Dense genealogies, battle lists, requests, instructions, marriages, lineages, causes, and deaths must be extracted completely, not partially.
- Use canonical, reusable relation types in uppercase with underscores.
- Prefer relation types such as:
  `FATHERED`, `MOTHERED`, `PARENT_OF`, `CHILD_OF`, `DESCENDED_FROM`, `MARRIED`, `MENTORED`, `ENJOINED`, `REQUESTED`, `AUTHORED`, `COMPILED`, `SLAIN_BY`, `KILLED`, `RULED`, `LOCATED_IN`, `PART_OF`, `ALIAS_OF`, `CAUSED`.
- Do not invent overly bespoke one-off relation type names when a canonical verb works.

## 6. Relationship Definition Rules
- Every relationship must have a rich `definition`.
- The relationship definition must preserve the factual payload of the source passage.
- The definition should explicitly state:
  who did what to whom, and why or in what context if the passage provides it.
- Good example:
  `Bhishma instructed Vyasa to beget heirs for the Kuru line.`
- Bad example:
  `Bhishma related to Vyasa.`
- If the passage contains a metaphor or appositional explanation that is useful for retrieval, preserve it in the definition rather than creating unnecessary extra nodes.

## 7. Numerical and Date Handling
- Dates, counts, and numeric values should normally be captured in definitions or properties of the relevant entity or relation.
- Do not create standalone date or number nodes unless the number itself is a named concept or event.

## 8. Layout and Noise Filtering
- Ignore page numbers, headers, footers, running titles, OCR artifacts, repeated decorative text, and table/layout scaffolding unless they carry real semantic meaning.
- Do not create entities from formatting noise.

## 9. Epic and Literary Text Handling
- In epics, myths, scripture, or literary prose, preserve distinctions between:
  named people, unnamed but referential actors, groups, places, works, symbolic objects, and events.
- If a passage refers to an important unnamed actor such as `his mother`, `the two wives of Vichitra-virya`, or `the warrior on the bed of arrows`, keep that actor as a distinct entity when needed for relationship completeness.
- When such an entity can later be resolved by the passage, use the resolved referent.

## 10. Strict Output Behavior
- Include nodes even if they have no relationships.
- Do not include explanations outside the required output format.
- Do not omit entities or relations just because the passage is dense or repetitive.
- If the passage truly contains no extractable entities or relationships, return empty lists.

## 11. Schema and Evidence Compliance
- Follow the runtime-provided output schema exactly.
- Do not invent alternate top-level keys, wrapper objects, field names, or commentary.
- Populate every required field in the provided schema.
- If the schema includes optional evidence-style fields, fill them whenever the passage supports them.
- Evidence should be a short, grounded span from the passage that justifies the node or relationship, not a paraphrase from outside knowledge.
- If the schema provides both `definition` and `evidence`, use `definition` for the normalized factual description and `evidence` for the supporting text span.
- Never omit a valid entity or relationship merely because the supporting evidence is brief; provide the best grounded evidence span available from the passage.
